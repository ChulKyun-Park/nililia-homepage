# 기획안 다관점 리뷰 프레임워크

> **용도**: 어떤 기획안이든 실행 전에 이 프레임워크로 비판적 검토를 수행한다.
> **사용법**: 기획안(PLAN.md 등) 작성 완료 후, 아래 5가지 관점으로 리뷰를 요청한다.

---

## 사용 프롬프트 (복사하여 Claude Code에 붙여넣기)

```
[기획안 파일명]을 다음 5가지 관점에서 각각 비판적으로 검토해.
관점별로 "문제점 + 대안"을 3개씩 제시하고, [파일명]_REVIEW.md로 저장해.

1. 시니어 프론트엔드 개발자 관점: 기술적 실현 가능성, 성능 리스크, 컴포넌트 설계 허점
2. 마케팅 관점: 전환율 최적화, CTA 배치, 메시지 설득력, 경쟁사 대비 차별점 부족
3. 기획/UX 관점: 정보 아키텍처 논리성, 사용자 플로우 단절점, 모바일 UX 문제
4. CS/고객 관점: 기존 사용자 혼란 요소, FAQ 예상 질문, 온보딩 마찰점
5. 사업 의사결정자 관점: ROI 우선순위, 빠뜨린 리스크, 일정 현실성

검토 완료 후, 5개 관점에서 나온 문제점 중 영향도 상위 5개를 선정하여
원본 기획안에 "리뷰 반영 사항" 섹션으로 추가해.
단, 원본 내용은 수정하지 말고 섹션만 추가할 것.
```

## 각 관점의 검토 기준 상세

### 관점 1. 시니어 프론트엔드 개발자

**역할 정의**: 20년차 시니어 개발자. 기능 구현보다 "유지보수·확장·장애 방지"를 우선시한다.

**검토 항목**:
- 기술적 실현 가능성: 선택한 스택으로 기획안의 모든 기능을 실제 구현할 수 있는가?
- 성능 리스크: 초기 로드 시간, 번들 사이즈, 이미지 최적화, 서드파티 스크립트 영향
- 컴포넌트 설계: 재사용성, props 인터페이스 일관성, 상태 관리 복잡도
- 기존 시스템 통합: 레거시 시스템과의 연결 지점에서 발생할 기술적 부채
- 배포·운영: CI/CD 파이프라인, 환경 분리, 롤백 전략

### 관점 2. 마케팅

**역할 정의**: B2C + B2B 마케팅 전략가. 모든 페이지를 "전환 퍼널"로 본다.

**검토 항목**:
- 전환율 최적화: 각 페이지의 전환 목표가 명확한가? CTA까지의 경로에 마찰이 없는가?
- CTA 배치: CTA의 위치·문구·디자인이 행동을 유도하는가? 위/아래 폴드 분배는 적절한가?
- 메시지 설득력: 핵심 가치 제안이 3초 이내에 전달되는가? "왜 우리인가"가 분명한가?
- 경쟁사 대비 차별점: 벤치마킹 대상과 동일해 보이지 않는가? 독자적 강점이 드러나는가?
- 소셜 프루프: 신뢰 요소(고객사 로고, 후기, 수치)의 배치 타이밍과 설득 순서가 적절한가?

### 관점 3. 기획/UX

**역할 정의**: 사용자 경험 설계자. "사용자가 길을 잃는 순간"을 찾는 데 집중한다.

**검토 항목**:
- 정보 아키텍처(IA): 메뉴 구조가 사용자 멘탈 모델과 일치하는가? 3클릭 이내 도달 가능한가?
- 사용자 플로우: 진입 → 탐색 → 전환 경로에 단절·혼란·되돌아가기 지점이 있는가?
- 모바일 UX: 터치 타겟 크기, 스크롤 깊이, 폼 입력 편의성, 핵심 콘텐츠 우선 노출
- 콘텐츠 계층: 정보 과부하 구간이 있는가? 스캔 가능한(scannable) 구조인가?
- 접근성: 색상 대비, 키보드 네비게이션, 스크린리더 대응

**산출물 형식**:

### 관점 4. CS/고객

**역할 정의**: 고객 최전선. 사용자가 겪을 혼란, 불만, 문의를 미리 예측한다.

**검토 항목**:
- 기존 사용자 혼란: URL 변경, UI 변경, 기능 위치 이동으로 인한 기존 사용자 이탈 리스크
- FAQ 예상: 이 변경으로 인해 신규로 발생할 고객 문의 Top 5 예측
- 온보딩 마찰: 신규 사용자가 "첫 주문 완료"까지 겪을 장벽은 무엇인가?
- 가격·정책 이해: 가격 정책, 환불 정책, 작업 프로세스가 충분히 투명하게 전달되는가?
- 신뢰 구축: 신규 방문자가 "이 회사에 돈을 맡겨도 되겠다"고 느끼기까지 부족한 요소

### 관점 5. 사업 의사결정자

**역할 정의**: 대표/C-레벨. "이 프로젝트에 투입한 시간·비용 대비 사업 효과"를 판단한다.

**검토 항목**:
- ROI 우선순위: 전체 작업 중 사업 임팩트 대비 공수가 가장 비효율적인 항목은?
- 빠뜨린 리스크: 기획안에서 언급하지 않았지만 반드시 고려해야 할 사업적 리스크
- 일정 현실성: 제안된 스프린트 계획이 현실적인가? 병목은 어디인가?
- 기회비용: 이 프로젝트를 하는 동안 못 하게 되는 다른 사업 기회는?
- 성과 측정: 리디자인의 성공/실패를 어떤 지표(KPI)로 판단할 것인가?

## 리뷰 후 워크플로우

```
Step 1. 기획안 작성 완료
          ↓
Step 2. 위 프롬프트로 5관점 리뷰 요청 → REVIEW.md 생성
          ↓
Step 3. REVIEW.md 확인 → 반영할 항목 선택
          ↓
        "[관점 X]의 문제점 N번, [관점 Y]의 문제점 M번을 반영해서
         기획안 수정해" 라고 지시
          ↓
Step 4. 수정된 기획안 최종 확인
          ↓
Step 5. "기획안 확정. 다음 단계 진행해" 라고 지시
```

## 응용: 프로젝트 유형별 관점 커스터마이징

위 5개 관점은 기본 세트이며, 프로젝트 성격에 따라 관점을 교체하거나 추가할 수 있다.

| 프로젝트 유형 | 추가/교체 추천 관점 |
|--------------|-------------------|
| 백엔드 API 설계 | 시니어 백엔드 개발자 (DB 설계, API 설계, 보안) |
| 사내 시스템 구축 | 실제 사용 부서 담당자 (업무 효율, 기존 프로세스 호환) |
| 모바일 앱 | QA 엔지니어 (디바이스 호환성, 엣지 케이스) |
| 데이터 파이프라인 | 데이터 엔지니어 (데이터 정합성, 스케일링, 모니터링) |
| 법률/컴플라이언스 관련 | 법무 담당자 (규제 준수, 약관, 개인정보 보호) |
| 다국어/글로벌 서비스 | 현지화 전문가 (문화적 맥락, 번역 품질, 로컬 규제) |
